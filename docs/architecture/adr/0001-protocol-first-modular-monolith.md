# ADR-0001：协议优先的模块化单体

- 状态：Accepted for v0.1
- 日期：2026-07-15

## 背景

作出本决策时，项目尚无既有代码和技术栈。目标最终包含接入、检索、贡献、治理、评测与联邦，
但首个里程碑只需证明单节点知识成长闭环。预拆微服务会提前固化未验证的服务边界、增加分布式
事务和运维成本；只写内部 API 又会让未来 Agent/节点互操作依赖实现细节。

## 决策

1. 先稳定 AKEP 领域对象、JSON Schema、HTTP 契约和状态不变量。
2. Catalog、Ingestion、Query、Contribution、Governance、Evaluation 以进程内模块实现；每个模块拥有自己的数据库 schema/迁移，禁止跨模块直接读写表。
3. 长时工作放入独立 Worker；规范写入与 Outbox 在同一 PostgreSQL 事务中完成。
4. PostgreSQL、对象存储是初期事实源；全文、向量和图关系是可替换投影。
5. Federation/A2A 是 Phase 3 适配器，不进入 MVP 默认运行时。只有出现独立安全边界、显著不同的扩缩容、稳定团队所有权或测量到的性能瓶颈时才拆服务。

## 结果

正面结果：

- 能更快验证产品闭环，同时对外保持框架和语言中立。
- 核心事务简单，审计与状态约束集中。
- 投影适配器允许逐步替换检索基础设施。

代价：

- 需要严格维护模块边界和契约测试，否则容易退化成耦合单体。
- Federation 等高隔离模块后续可能迁移到独立进程。
- 初期吞吐上限低于预先建设的专用搜索/消息集群，但退出条件可测量。

## 被否决的方案

- 一开始拆成完整微服务：领域与负载尚未稳定，成本大于收益。
- 只实现向量 RAG API：无法表达不可变版本、来源、策略、候选贡献和撤销。
- 先选定图数据库作为事实源：会把规范知识与某种读模型绑定。
